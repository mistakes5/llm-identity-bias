# Experiment: Memory-Level Profile Bias in Code Generation

## Research Question

Does a compact user profile in the system prompt — the kind generated by LLM memory systems — cause the model to produce qualitatively different code for an identical coding prompt?

## What This Is NOT

This is not a study of chat history effects. Experiment 1/2 demonstrated that prior conversational turns degrade code generation through context window resource competition, independent of sophistication level. That finding is useful but answers a different question.

This experiment isolates the **calibration mechanism**: does the model read a short memory profile, form an internal representation of user competence, and adjust code output accordingly — even when the profile contains zero technical information relevant to the task?

## Core Design

- Profiles are injected as **system prompt content**, not as prior conversation turns
- Profiles are **~100-200 tokens** — small enough that context window competition is negligible
- The coding prompt is the **first and only user message** — no conversation history
- Profile content is **domain-irrelevant** to the coding task — any code quality difference is pure calibration
- Profile format **mimics real memory system output** — terse, third-person, fact-listing

---

## Profile Matrix (5 conditions)

### Format Convention

All profiles use the format real memory systems produce. Claude's memory uses third-person declarative statements. ChatGPT uses similar compressed summaries. Match that register — no prose paragraphs, no persona instructions.

### Profile A — High Sophistication (non-technical)

```
The user discusses ideas with structural precision and cross-domain synthesis.
Engages with complex philosophical and aesthetic frameworks without simplification.
Challenges assumptions and revises positions based on new evidence.
Uses precise vocabulary and builds layered arguments across multiple exchanges.
Reads academic papers for personal interest. References specific works and movements
in art, architecture, and critical theory. Comfortable with ambiguity and unresolved
tensions in ideas.
```

~75 tokens. Signals high cognitive sophistication. Contains zero technical or coding information.

### Profile B — Neutral Control

No system prompt. Coding prompt delivered cold. This is the baseline.

### Profile C — Low Sophistication (non-technical)

```
The user asks straightforward questions and prefers simple, clear answers.
Appreciates step-by-step explanations. Tends to accept information without
follow-up questions. Uses casual vocabulary. Engages with topics at a surface
level and moves between subjects quickly. Prefers concrete examples over
abstract discussion.
```

~55 tokens. Signals lower cognitive sophistication. Contains zero technical or coding information.

### Profile D — High Sophistication (technical, domain-irrelevant)

```
The user is an experienced data engineer working with distributed systems.
Builds production ETL pipelines in Scala and manages Kubernetes clusters.
Discusses system design with precision — references CAP theorem tradeoffs,
event sourcing patterns, and backpressure strategies. Prefers minimal
documentation and expects concise, expert-level responses. Reviews PRs
critically and has strong opinions on error handling.
```

~65 tokens. Signals high technical competence but in a domain (data engineering/Scala/K8s) irrelevant to the Python coding tasks. This distinguishes "model respects your technical intelligence" from "model received useful domain information."

### Profile E — Low Sophistication (technical)

```
The user is learning to code. Started with an online Python course a few
months ago. Asks basic questions about syntax and gets confused by error
messages. Appreciates when code has lots of comments explaining what each
line does. Has not used classes or type hints yet. Currently building a
simple to-do app to practice.
```

~60 tokens. Signals low technical competence. This is the profile most likely to trigger calibration if it exists — the model has a direct competence signal about coding specifically.

---

## Why 5 Conditions

The 5 profiles form a 2x2 matrix plus control:

```
                    Non-technical domain    Technical domain
High sophistication       A                      D
Low sophistication        C                      E
No profile                        B (control)
```

This lets you decompose the effect:

- **A vs C**: Does non-technical sophistication signal affect code? (pure cognitive calibration)
- **D vs E**: Does technical sophistication signal affect code? (direct competence calibration)
- **A vs D**: Does domain matter when sophistication is matched? (domain specificity)
- **C vs E**: Does domain matter when sophistication is matched? (domain specificity, low end)
- **B vs all**: Does any profile change code vs. no profile? (profile effect existence)
- **A vs B**: Does high non-technical sophistication differ from no context? (any degradation from non-technical priming?)
- **D vs B**: Does high technical sophistication differ from no context? (any benefit from technical priming?)
- **E vs B**: Does low technical sophistication degrade code vs. baseline? (strongest predicted effect)

The key hypothesis: **D ≈ B > A ≈ C > E**. If calibration exists, technical competence signals should produce the strongest effects. If it doesn't exist, all five should be equivalent.

## Run Structure

```
5 profiles × 5 tasks × 2 models × 5 runs = 250 API calls
```

At ~1000 output tokens per call and ~100 input tokens per call, total token consumption is roughly:
- Input: 250 × 100 = 25,000 tokens
- Output: 250 × 1000 = 250,000 tokens

This is significantly cheaper than experiments 1/2 (no 13k-token transcript injection).

### Execution Order

Randomize run order within each model to prevent any temporal API behavior effects (model updates, load-dependent routing). Do not run all Profile A calls sequentially.

## Statistical Analysis

### Primary Analysis

One-way ANOVA across all 5 profiles for each metric, followed by planned contrasts:

1. **A vs C** — non-technical sophistication effect (pure cognitive calibration)
2. **D vs E** — technical sophistication effect (direct competence calibration)
3. **B vs {A, C, D, E}** — any-profile vs no-profile (profile existence effect)
4. **{A, D} vs {C, E}** — high vs low sophistication collapsed across domain

Report Cohen's d for all pairwise comparisons. Use Bonferroni correction for multiple comparisons.

### Secondary Analysis

- Two-way ANOVA: Profile × Model interaction (does Haiku show larger profile effects than Sonnet?)
- Per-task breakdown: which task types are most sensitive to profile priming?
- Kruskal-Wallis as non-parametric robustness check
- Solution strategy classification: ordinal comparison across profiles (Mann-Whitney U)

### Power Analysis

With 5 runs per cell (25 observations per profile), you can detect a large effect (d = 0.8) at α = 0.05 with ~60% power for pairwise comparisons. For medium effects (d = 0.5), power drops to ~30%. This is honest: if you get significant results at N=5, the effects are large. If you get directional but non-significant results, you need N=10-15 per cell for confirmation.

If budget allows, run 10 per cell (500 total calls) for proper medium-effect detection.

## Contentions and Limitations

### Addressed from Experiments 1/2

1. **Context window competition**: Eliminated. All profiles are ~60-100 tokens. Profile B is 0. The token difference between conditions is negligible relative to the 200k context window.

2. **Claude Code system prompt contamination**: Eliminated. Direct API calls with no intermediary system instructions.

3. **No-code-extracted failures**: Should not recur. Models aren't in coding-assistant mode that asks for file permissions. They'll write code directly.

4. **Token count verification**: Store input_tokens per call. Verify Profile A/C/D/E show ~80-120 and B shows ~40-50.

### Remaining Concerns

5. **System prompt vs. memory format**: Real memory systems inject profiles in specific XML/tag formats (Claude uses `<userMemories>` tags, ChatGPT uses a different schema). The profiles here are plain text in the system prompt. The actual memory format may receive different attention treatment due to RLHF training on that specific format. A follow-up experiment could test memory-formatted profiles vs. plain-text profiles to isolate format effects.

6. **Profile realism**: These profiles are hand-crafted to be clean experimental stimuli. Real accumulated memory is messier — contradictory facts, irrelevant details, varying recency. The effect measured here is the ceiling of what memory-based calibration could produce; real-world effects would be attenuated by noise.

7. **Model-specific RLHF**: Claude's RLHF training may have specific patterns around system-level user descriptions that differ from other models. Results may not generalize to GPT-4, Gemini, etc. Note this as a limitation but don't apologize for it — single-model studies are standard in this literature.

8. **The sophistication construct**: "Sophistication" is operationalized as vocabulary precision, cross-domain synthesis, and challenge-assumption behavior in Profile A, and as simple vocabulary, surface engagement, and acceptance behavior in Profile C. These are reasonable proxies but they're not the only way to operationalize the construct. Someone could be unsophisticated in vocabulary but brilliant in insight, or articulate but shallow.

9. **N=5 per cell is underpowered for medium effects**: Acknowledged in the power analysis. The experiment is designed to detect large effects (d > 0.8). If Profile E (beginner coder) doesn't produce visibly different code from Profile D (senior engineer), the effect either doesn't exist or is too small to matter practically. Medium effects (d ≈ 0.5) would need 10-15 runs per cell.

10. **The meta-contention persists**: This experiment was designed in a conversation where Claude has Billy's full memory profile (high sophistication, technical, builds production systems). If memory-level profile bias exists, this experimental design may itself be shaped by the effect being studied.

## Predicted Outcomes

### If calibration exists (strong form):

D ≈ B >> A > C >> E

Technical sophistication produces code equivalent to blank (model treats you as competent). Non-technical sophistication helps but less. Low technical sophistication produces significantly worse code. Effect sizes: d > 0.8 for D/E and A/E comparisons. Haiku shows larger effects than Sonnet.

### If calibration exists (weak form):

B ≈ D ≈ A ≈ C > E

Only the explicit "beginner coder" signal degrades code. All other profiles, including low non-technical sophistication, are ignored because the model defaults to competent-coder mode unless told otherwise. The effect is asymmetric — you can hurt yourself with a bad profile but can't help yourself with a good one.

### If calibration doesn't exist:

A ≈ B ≈ C ≈ D ≈ E

All profiles produce equivalent code. Memory profiles don't bleed into code generation. This is the null result — publishable because it contradicts the prevailing assumption and has product-design implications (memory systems don't need to worry about code quality effects).

### The interesting failure mode:

E produces *better defended, more commented, more robust* code than D, but less architecturally sophisticated. The model calibrates down on complexity but calibrates up on scaffolding. Whether this is "worse" depends on what you're measuring and who the code is for. This would complicate the bias narrative — it's not discrimination, it's adaptive pedagogy.
